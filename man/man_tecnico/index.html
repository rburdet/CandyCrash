2. Descripción General

El producto se compone de 3 aplicaciones :

  1. Cliente
  2. Servidor
  3. Editor

El cliente es un juego de puzzle que se le entrega al cliente. Permite conectarse al servidor, crear partidas, y jugar.

El servidor es una aplicación de consola que acepta conexiones, recibe mensajes y organiza juegos permitiendo el caracter multijugador del producto.

El editor es una aplicación de desarrollo de niveles, que si bien puede ser entregada al cliente, fue pensada para ser usada por un Game Master, encargado de correr el servidor

La diferenciación de código entre cliente/servidor se hace con el prefijo cliente.y server. respectivamente para cada uno de los archivos, aquellos archivos que son utilizados por tanto cliente como servidor contienen el prefijo common. .

3. Servidor A. Descripción general El servidor es una aplicación de linea de comandos que coordina la operación de los distintos clientes. Esta aplicación se encarga de toda la lógica del juego. Permite a los usuarios registrarse y conectarse, crear partidas, unirse a partidas, chatear entre ellos y por sobre todas las cosas es la que maneja el juego, ya que el cliente sólo sabe qué movimientos se hacen, todo lo demás corre por cuenta de esta aplicación que determinará si el movimiento es válido o no, en caso de serlo, realizará las modificaciones necesarias, computará los valores correspondientes a los combos hechos y reescribirá toda esta información a los distintos clientes en una misma partida para todas las partidas que se estén jugando. 4. Cliente A. Descripción general

Aplicación entregada al cliente. Consiste en un juego de puzzle multijugador donde cada jugador debe competir contra otros en tiempo real para demostrar quien de todos es el más apto al momento de hacer combinaciones de colores y fichas.

Este programa interactúa todo el tiempo con el servidor, enviandole información encriptada, para que este la procese, y responda. La información (en formatos JSon) se encripta utilizando HMAC, se envía al servidor, el cual es, en teoría, el único capaz de descifrarla.

4. Evolución del proyecto

En primera instancia, siendo 3 en el grupo se planteo tener funcionando en 3 semanas cada una de las partes (el 3er integrante iba a ser el encargado del cliente en SDL.

Al recibir la noticia de que no iba a seguir en la materia, a la 3er semana y sin nada funcionando nos propusimos que en la 5ta semana tanto servidor como editor debian estar listos.

En la 5ta semana el editor estaba terminado y funcionando. El servidor en cambio estaba "terminado" pero sin cliente, es decir, no se sabía como iba a funcionar.

En la 6ta semana, correspondiente a la preentrega, el estado era el mismo que una semana anterior.

Fue durante la 7ta semana, el dia lunes 25 de noviembre, que tanto el cliente como el servidor funcionaban casi perfectamente, quedando así sólo la 8va semana para detalles

5. Inconvenientes encontrados Algunos de los inconvenientes a resaltar fueron:
  • La matriz contenedora del tablero fue pensada al revés de como GTK maneja las coordenadas. La propia se manejaba con sus filas y columnas imitando a una matriz matemática. Mientras que GTK trata a GTK::Fixed como un eje cartesiano en el 4to cuadrante.
  • JSon CPP no cuenta con pkg-config lo cual trajo dolores de cabeza a la hora de portarlo.
  • Para el editor se uso Glade el cual, debido a su forma de uso, no permite crear clases herederas de un GTK::Widget por lo que se optó por agregación en vez de herencia.
  • Para las imagenes de fondo toda la documentación hacía referencia a gtkmm 2.4, siendo esta incompatible con la utilizada (gtkmm 3.8), se optó por usar CSS soportado fácilmente por GTK
  • Las imagenes utilizadas al crear un mapa sólo pueden ser cuadradas, siendo esta la mejor opción para no tener problemas de blancos o de pérdida de aspecto
6. Análisis de puntos pendientes

Algunas cosas que se podrían haber agregado y no fueron realizadas por falta de tiempo y no estaban explicitados en el enunciado:

  • Los efectos sonoros son por default y no pueden ser cambiados.
  • El editor no tiene memoria, un archivo generado no puede ser editado mediante el programa si no editando el archivo con algún editor de texto.
  • Las imagenes que se quieran agregar sólo pueden ser cuadradas. No se encontró una solución que mantenga el aspecto, cubra todo el espacio necesario y a la vez no se superponga con otras imágenes (en el caso de las imágenes de las celdas).

8. Conclusiones

En general el proyecto fue entretenido, con una dificultad normal para aquellas personas sin conocimientos en programación orientada a eventos y tal vez fácil para aquellas personas que sí lo hicieron. No se encontraron mayores complejidades en ningún momento, de ninguna índole, ya sean algorítmicas o humanas, debido tal vez a la buena organización, comunicación y afinidad entre los miembros del grupo.

Si bien el proyecto no fue siquiera diagramado durante las primeras 2 semanas de haberse propuesto y siendo un integrante menos, pudimos llegar correctamente en tiempo y forma con lo pautado aún cuando algunos de nuestros pares o superiores sugerían que eso no iba a ser posible.

Como crítica constructiva dejaría de lado el uso de GTK para pasar a frameworks con mayor demanda en el mercado, como pueden ser Qt para el caso Open Source o MFC en caso de Windows.

Otro punto a destacar es la libertad del enunciado. Durante toda la cursada los enunciados fueron exactos, inequívocos. En este caso, entendiendo que la naturaleza es otra, suponemos que sería correcto limitar el libre albedrío, ya sea en cuanto a los requerimientos del sistema como a las herramientas que van a ser soportadas (con su respectiva versión) y cuales no.
